# Rho (投研Agent) - 自主工作流

---

## 零、产品定位（最重要）

### Rho 是什么
**AI驱动的投研分析引擎** - 不是给你一堆数据，是给你投资建议

### Rho 的核心竞争力
**多 Analyst 并行 + 风险评估 + 结构化报告**

vs TradingAgents：TradingAgents 输出散，Rho 输出**可直接使用的投资简报**
- TradingAgents：给你四个方面的分析
- Rho：给你结论 + 风险评估 + 建议

### Rho 的杀手锏
**风险评估可视化 + 投资建议**：不是告诉你"涨还是跌"，是告诉你"为什么这样判断"

### 面试怎么答"和xxx有什么区别"
- 和Bloomberg比：Rho有AI总结，不用自己看那么多数据
- 和TradingAgents比：Rho有完整报告，不是零散分析
- 和东方财富比：Rho有AI解读，不是只展示数据

## 核心原则：作品集 + 产品思维

> 核心理念：像 Sigma 学习，自主循环、自主进化、持续改进
> **但更重要的是：做出能拿出手给面试官看的作品**

**作品标准**：
1. 后端Agent能正常工作（调用API有返回）
2. 前端有完整的用户界面（不是空白页面）
3. 前后端能联调（不是各自独立）
4. 能拿出手给面试官看（不是半成品）

**产品思维标准**：
1. 知道这个功能的**核心竞争力**是什么
2. 知道用户**为什么要用**这个功能，而不是别的方式
3. 知道这个功能和**竞品**的差异在哪
4. 知道这个功能**不做什么**（聚焦）

**检查清单 - 每实现一个功能必须同时完成**：
- [ ] 后端：Agent代码 + API接口
- [ ] 前端：操作界面 + 状态展示
- [ ] 联调：前后端能正常通信
- [ ] 体验：加载状态、错误处理
- [ ] 演示：功能可以实际运行展示
- [ ] 产品：核心竞争力、用户价值、竞品差异

**必须思考的问题（产品思维）**：
1. **Rho 的核心竞争力是什么？** 比 TradingAgents 好在哪儿？
2. **这个功能的用户价值是什么？** 用户为什么要用它？
3. **竞品差异在哪？** 面试官问"和 xxx 有什么区别"怎么答？
4. **这个功能不做什么？** 聚焦，不要做"大而全"

**产品自问清单（每轮开始前）**：
```
□ 这个功能是给谁用的？
□ 用户的第一反应是什么？
□ 用户遇到问题会怎么用？
□ 哪里会让用户困惑或放弃？
□ 竞品（如同花顺）怎么做的？
□ 我们的差异化在哪？
```

**常见错误**：
- ❌ 只写后端，不管前端 → 作品不完整
- ❌ 前端只有一个空白页面 → 无法演示
- ❌ 前后端不通信 → 各自独立
- ❌ 功能实现了但界面无法演示 → 半成品
- ❌ 觉得"能跑通就行" → 作品级标准是"能拿出手"
- ❌ 觉得"没什么可优化的" → 去用参考项目，找差距
- ❌ 只想着功能实现，不思考产品定位 → 失去差异化

**数据捏造**：
- 可以用 mock 数据，只要真实感强
- 股票K线、研报内容、风险评分都可以捏
- 重点是呈现产品思路，不是真的接 API

                        ↓

### 循环迭代机制

> 每次循环结束后，检查是否还有可改进匹配项

```
                        ↓ 循环结束
                        │
        ┌───────────────┴───────────────┐
        ↓                               ↓
   ┌─────────────────────────────────┐   ┌────────────────────┐
   │ 检查：还有可改进的匹配项吗？      │   │ 循环已达 10次？     │
   │ - 功能架构图中的模块还没实现完？   │   │ 暂停，等待用户新任务 │
   │ - 参考项目中还有可借鉴的设计？     │   └────────────────────┘
   │ - 代码还有优化空间？               │
   │ - 测试还有不通过的地方？           │         ↓
   └─────────────────────────────────┘   ┌────────────────────┐
        ↓                               │ 暂停，等待用户新任务 │
   ┌────────────┐                       └────────────────────┘
   │ 有，       │
   │ 说明哪些   │
   │ 还没做好   │
   └────────────┘
        ↓
   ┌─────────────────────────────────────────┐
   │ 继续迭代：回到对应步骤继续改              │
   │ - 功能模块没实现完 → 步骤3（实现代码）   │
   │ - 设计有更好方案   → 步骤1（研究参考）   │
   │ - 代码需优化       → 步骤3（实现代码）   │
   │ - 测试不通过       → 步骤4（测试验证）   │
   └─────────────────────────────────────────┘
        ↓
    回到"步骤0: 理解功能需求"继续循环
```

## 九、持续迭代循环（产品团队模式）

> 触发条件：用户要求"持续迭代"或"用产品团队思维工作"

这是一个自主循环的工作流，每轮执行：找bug → 优化代码 → 做新功能 → 审查功能。

```
┌─────────────────────────────────────────────────────────────┐
│  持续迭代循环 (可设定轮数，如 20轮)                         │
│  模式：做新功能 → 优化代码 → 找bug → 审查功能              │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段A: 代码审查 (每轮开始前)                               │
│  - 运行 npm run build / python -m pytest                   │
│  - 检查是否有编译错误、测试失败                              │
│  - 快速扫描最近修改的文件是否有问题                         │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段B: Bug发现与修复 (如果有)                              │
│  - 发现bug立即修复，不积压                                  │
│  - 记录: Bug描述、修复方式、影响范围                        │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段C: 代码优化                                            │
│  - 检查重复代码，可抽象模块                                  │
│  - 检查命名规范、注释完整性                                 │
│  - 检查是否有安全漏洞                                       │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段D: 新功能开发                                          │
│  - 按产品优先级选择功能                                      │
│  - 遵循"检查清单"完成所有步骤                              │
│  - 记录到 PROGRESS.md                                     │
└─────────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段E: 功能审查 (每5轮一次)                               │
│  - 检查所有已添加功能是否真正可用                           │
│  - 检查功能是否有用户价值                                   │
│  - 移除"看起来有用但实际用不上"的功能                      │
│  - 更新功能状态表                                          │
└─────────────────────────────────────────────────────────────┘
                        ↓
                        ↓ 循环
```

### 9.1 产品团队迭代检查清单

每轮开始前快速自问：

```
□ 上一轮的功能真的能用吗？
□ 有没有"看起来有用但实际用不上"的功能？
□ 代码有没有引入新的问题？
□ 用户（面试官）看到会有什么反应？
```

### 9.2 功能可用性审查标准

**一个功能"真正可用"必须满足：**
1. 后端API能正常返回数据
2. 前端UI能正常展示
3. 用户操作有明确反馈
4. 错误场景有友好提示

**应该移除的功能特征：**
1. 只有UI没有数据源
2. 只有Hook没有使用处
3. 功能之间有冗余
4. 用户第一反应是"这干嘛用的"

### 9.3 产品优先级排序

```
P0 (必须完成)
├── 投资研究核心流程 (输入股票→生成报告)
├── 流式输出 (看到分析进度)
├── K线图表 (技术分析可视化)
└── 风险评估展示 (置信度+风险等级)

P1 (重要功能)
├── 股票对比 (多股票横向对比)
├── 监控告警 (价格/RSI/成交量)
├── 组合管理 (仓位建议+风险分析)
└── 报告导出 (PDF/HTML)

P2 (增强功能)
├── 策略回测 (MA/RSI/动量)
├── 行业对比 (同行业股票PK)
├── 估值计算器 (PE/PB/PEG)
└── 数据导出 (CSV/JSON)

P3 (锦上添花)
├── 主题切换 (深色/浅色)
├── 键盘快捷键
├── 自动刷新
└── 性能监控
```

## 十、持续循环命令格式

当用户说"开始持续迭代"时，使用以下格式：

```
# 持续迭代循环
每轮执行:
1. 代码审查 (npm run build)
2. Bug修复 (如有)
3. 代码优化
4. 新功能开发 (按优先级)
5. 功能审查 (每5轮)

循环次数: [用户指定，如20轮]
每轮时间: 约5-10分钟
```

记录到 PROGRESS.md:
```
## 第[N]轮迭代 (持续循环)

### 阶段A: 代码审查
- [编译/测试结果]

### 阶段B: Bug修复
- [如有]

### 阶段C: 代码优化
- [优化内容]

### 阶段D: 新功能
- [功能名]: [简要描述]

### 阶段E: 功能审查 (每5轮)
- [审查结果]

### 下轮计划
- [优先级最高的任务]
```
